Cómo funciona la API Inmobiliaria

Esta API recibe datos de inmuebles en formato JSON, valida la entrada con Pydantic, aplica limpieza opcional con un módulo propio, calcula estadísticas con NumPy y guarda los resultados en memoria para consultarlos con endpoints CRUD.

Dominio: mercado inmobiliario
Validación: Pydantic
Cálculo: NumPy
API: FastAPI
Servidor: Uvicorn

Flujo completo del request

Esta es la secuencia principal del endpoint POST /analizar. La idea es mostrar no solo el código, sino el recorrido lógico de los datos dentro del sistema.

1
Cliente envía JSON El usuario envía un conjunto de inmuebles con nombre del análisis, moneda y opción de limpieza.
2
FastAPI recibe el request El endpoint /analizar recibe el cuerpo de la solicitud y lo pasa al modelo principal.
3
Pydantic valida Se controlan tipos, rangos y restricciones como area_m2 > 0 y longitud mínima.
4
Limpieza y análisis Si corresponde, se ejecuta DataCleaner. Luego analizar_inmuebles calcula métricas.
5
Respuesta JSON El resultado se guarda en historial y se responde al cliente con un JSON estructurado.

Estructura del proyecto y orden de presentación

Para explicarle el proyecto al profesor, conviene seguir el mismo orden lógico con el que se procesa un request: primero las dependencias, luego la carpeta api/, después analysis/, luego limpieza/, después los datos del dominio y, al final, los módulos complementarios.

1. requirements.txt

Este archivo se muestra rápido para señalar las dependencias principales del proyecto.

  • FastAPI → framework para construir la API
  • Uvicorn → servidor ASGI para ejecutar la aplicación
  • Pydantic → validación de datos
  • NumPy → cálculos estadísticos
  • Pandas → manipulación de datos tabulares

Qué decir: “Aquí están las dependencias principales del proyecto. FastAPI para la API, Pydantic para validación, NumPy para estadísticas, Pandas para manipulación de datos y Uvicorn para ejecutar el servidor.”

Posible pregunta: “¿Por qué usaste NumPy y no solo Python?” Respuesta: porque facilita cálculos estadísticos vectorizados y más claros.

2. api/ → analysis_schemas.py

La carpeta api/ se muestra primero porque ahí se ve qué entra a la API, cómo se valida y cuál es el endpoint principal.

  • InmuebleInput → representa cada inmueble individual
  • AnalisisInmobiliarioInput → cuerpo principal del endpoint /analizar
  • AnalisisInmobiliarioResult → salida con métricas calculadas
  • AnalisisGuardado → estructura persistida en historial

Qué destacar: uso de Field, Optional, tipos distintos y validaciones como min_length, gt, ge y le.

Posible pregunta: “¿Por qué area_m2 > 0?” Porque evita áreas imposibles y protege el cálculo de precio_m2.

3. api/ → main.py

Después se muestra la implementación de la API y el flujo de endpoints.

  • Imports principales
  • app = FastAPI(...)
  • historial y contador_id
  • /limpiar
  • /analizar
  • /historial
  • /historial/{id}
  • DELETE /historial/{id}

Qué decir sobre /analizar: “Este es el endpoint principal. Recibe un AnalisisInmobiliarioInput, valida los inmuebles con Pydantic, opcionalmente aplica limpieza, luego llama la función analizar_inmuebles, construye la salida con AnalisisInmobiliarioResult y guarda el resultado en memoria.”

Qué remarcar: uso de HTTPException(404), separación entre validación, lógica y persistencia temporal, y almacenamiento en memoria con diccionario.

4. analysis/ → stats.py

Aquí el foco deja de ser la API y pasa a la lógica estadística del proyecto.

  • Función _r4 para redondeo a 4 decimales
  • Función principal analizar_inmuebles
  • Uso de numpy
  • mean, median, min, max, std
  • ddof=1
  • Cálculo de precio_m2

Qué decir: “Aquí separé la lógica de procesamiento del framework. Esta función no depende de FastAPI. Recibe una lista de inmuebles ya validados, convierte los datos a arreglos NumPy y calcula métricas sobre precio, área y precio por metro cuadrado.”

Posible pregunta: “¿Por qué ddof=1?” Porque estoy usando desviación estándar muestral, no poblacional.

5. limpieza/

Esta carpeta se muestra después de analysis/, como módulo adicional que mejora la calidad de los datos antes del análisis.

  • schemas.py → esquema de configuración del proceso y esquema del reporte
  • pipeline.py → estandarización de columnas, vacíos a NaN, duplicados, limpieza monetaria, conversión numérica segura e imputación de nulos
  • cleaner.py → clase DataCleaner para reutilizar el pipeline desde la API
  • __init__.py → expone la API pública del paquete

Qué decir: “El módulo de limpieza no era estrictamente el mínimo del CRUD, pero lo integré para mejorar la calidad de los datos antes del análisis.”

6. Datos y módulos complementarios

  • inmuebles_bogota.csv → dataset base del dominio
  • data_limpia.csv → dataset resultante del preprocesamiento
  • eda/ → análisis exploratorio previo para entender variables y contexto
  • scripts/ → apoyo al flujo general del proyecto

Qué decir sobre los datos: “Estos archivos representan el origen del dominio. inmuebles_bogota.csv es el dataset base y data_limpia.csv refleja el resultado del preprocesamiento.”

eda/ y scripts/ se muestran solo al final, porque no son el centro del laboratorio de FastAPI.

Validación con Pydantic

Antes de procesar datos, la API valida cada inmueble. Esto evita que lleguen entradas inválidas a la lógica estadística y protege cálculos como precio_m2.

Ejemplos de validación

  • area_m2 debe ser mayor que 0.
  • valor debe ser numérico y mayor que 0.
  • habitaciones y banos tienen límites.
  • tipo exige longitud mínima.
  • La lista de inmuebles debe tener al menos 2 registros.

Impacto técnico

Si se enviaran datos como área negativa, textos en campos numéricos o una lista insuficiente de inmuebles, FastAPI devolvería un 422 antes de ejecutar NumPy. Esto mantiene la integridad del análisis.

Endpoints principales

Aquí se resume cómo se usa la API. Cada tarjeta tiene pestañas para ver el request, la explicación y la evidencia visual.

POST /analizar
Request exitoso 
{
  "nombre_analisis": "analisis valido",
  "inmuebles": [
    {
      "tipo": "apartamento",
      "area_m2": 60,
      "valor": 240000000,
      "habitaciones": 2,
      "banos": 1,
      "barrio": "Chapinero"
    },
    {
      "tipo": "casa",
      "area_m2": 120,
      "valor": 480000000,
      "habitaciones": 4,
      "banos": 3,
      "barrio": "Cedritos"
    }
  ],
  "moneda": "COP",
  "usar_limpieza": true
}
Este request pasa correctamente por Pydantic, luego la API ejecuta el análisis estadístico con NumPy, calcula métricas como promedio, mediana, mínimo, máximo y desviación estándar, y finalmente guarda el resultado en memoria dentro de historial.
POST /analizar
Error 422 
{
  "nombre_analisis": "error_area_negativa",
  "inmuebles": [
    {
      "tipo": "apartamento",
      "area_m2": -50,
      "valor": 250000000,
      "habitaciones": 3,
      "banos": 2,
      "barrio": "Chapinero"
    },
    {
      "tipo": "casa",
      "area_m2": 120,
      "valor": 450000000,
      "habitaciones": 4,
      "banos": 3,
      "barrio": "Cedritos"
    }
  ],
  "moneda": "COP",
  "usar_limpieza": true
}
Esta prueba produce HTTP 422 porque area_m2 tiene el valor -50 y el modelo exige que sea mayor que 0. La validación ocurre antes de ejecutar la lógica, por lo que la API bloquea la entrada inválida y evita resultados incoherentes.
GET /historial
Consulta de análisis 
GET /historial
Este endpoint no recibe body JSON. Su función es retornar todos los análisis guardados en memoria. Si previamente se ejecutó POST /analizar, aquí deben aparecer los resultados almacenados.
GET /historial/99
Error 404 
GET /historial/99
Esta prueba consulta un ID inexistente. La API busca el análisis en historial y, como no lo encuentra, lanza HTTPException(status_code=404). Esto demuestra manejo explícito de errores para recursos inexistentes.
DELETE /historial/{id}
Eliminación 
DELETE /historial/1
Este endpoint elimina un análisis previamente guardado. Si el ID existe, la API responde correctamente y elimina el registro en memoria. Si el ID no existe, también puede devolver 404.

Esta parte resume el valor académico de la evidencia mostrada en Swagger.

POST exitoso

Prueba que la API recibe datos válidos, ejecuta la lógica con NumPy y genera una respuesta estructurada.

POST con 422

Prueba que Pydantic protege el sistema y evita que la lógica procese entradas inválidas.

GET /historial

Prueba que existe persistencia temporal y que el CRUD puede consultar recursos almacenados.

Prueba el manejo correcto de errores cuando se intenta acceder a un recurso inexistente.

DELETE /historial/{id}

Prueba que la API elimina correctamente un análisis guardado y actualiza el historial.

Módulo de limpieza

Muestra que el proyecto no se limita al CRUD: también incorpora una etapa de calidad de datos previa al análisis.

Cierre del proyecto

Esta API demuestra integración entre validación, lógica estadística y exposición de servicios REST. La arquitectura modular facilita mantenimiento, pruebas y explicación académica.

Fortalezas

  • Dominio propio: análisis de inmuebles.
  • Validaciones con Pydantic y manejo de errores.
  • Lógica separada de FastAPI.
  • Uso de NumPy con métricas del dominio.
  • CRUD funcional en memoria.
  • Módulo extra de limpieza de datos.

Posible mejora futura

La siguiente evolución natural sería reemplazar el diccionario en memoria por una base de datos como PostgreSQL o MongoDB para mejorar persistencia y escalabilidad.

Consejo para exponer: primero muestra la idea general, luego el flujo del request, después esta estructura del proyecto, luego abre Swagger y usa las pruebas para demostrar validación, análisis y CRUD.
